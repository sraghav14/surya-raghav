Building a Website with AI: What I Learned from Using Claude as a Dev Partner

Surya Raghav February 2026 · 7 min read

I am a Finance and Data Science student. I have never taken a web development course. I know what HTML stands for, and I have a vague memory of messing around with CSS in middle school to customize a Tumblr page. So when DATA 305 — Generative AI for Problem Solving — asked us to build and deploy a full personal website using GitHub Pages, I had one thought: this is what AI was made for.

This post is a reflection on that process: how I used Claude as a development partner, what surprised me, where things broke down, and what I'd do differently next time.

Phase 1: Choosing a Framework (The Part Nobody Talks About)

Before a single line of code gets written, you have to make a decision that shapes everything else: what tool are you using? The assignment asked us to evaluate at least three static site generators — Jekyll, Hugo, Astro, Quarto, plain HTML, and others.

I started by asking Claude to walk me through the trade-offs. Not to make the decision for me, but to help me understand what questions I should even be asking. This felt like the natural first prompt:

"I'm building a personal portfolio site on GitHub Pages. I have basically no web dev experience. Can you compare Jekyll, Hugo, and plain HTML/CSS for a beginner — focusing on ease of setup, GitHub Pages compatibility, and how well you (Claude) know each one?"

The response was illuminating. Claude made a point I hadn't considered: its own knowledge matters. Jekyll, being older and more widely documented, had stronger AI support. Astro was cutting-edge but trickier to debug with AI assistance because the syntax was less common in training data.

I ended up going with plain HTML/CSS/JS. Here's why: I wanted full control. I wasn't trying to learn a framework — I was trying to ship a site. And for a beginner using AI, plain HTML means every line of code is something I can actually read and understand, rather than configuration files I'm copying blindly.

"The best tool is the one you can debug when things go wrong." — advice that applies whether you're writing code or making decisions in finance.

Phase 2: Building — Where AI Really Shines

Once I had a direction, the build phase was remarkably smooth. Claude was most useful when I gave it specific, context-rich prompts rather than vague ones. Compare these two:

❌ "Make me a portfolio website"
✅ "Create an HTML hero section with a two-column layout: left side has my name, a short bio, two CTA buttons, and three stat counters. Right side has a profile photo with a gradient border. Use CSS variables for a dark navy blue color theme. All text in Inter font."

The second prompt got me 90% of the way there in a single response. The specificity of the layout, the color theme, and the font choice meant I wasn't rewriting from scratch five times.

Key things I learned about prompting for code:

  • Describe the visual outcome, not just the technical goal. "I want a card that floats up 3px on hover with a blue glow" is better than "add hover effects."
  • Reference what you already have. Pasting in existing CSS and asking "add a responsive navbar that matches this style" produces far better results than starting fresh.
  • Ask for explanations. When Claude gave me complex CSS I didn't understand, asking "explain what this `backdrop-filter` property does" turned each bug fix into a mini lesson.
  • Use it to debug. When my sticky sidebar wasn't working on mobile, pasting the broken code and asking "why isn't `position: sticky` working here?" was faster than any Stack Overflow search.

Phase 3: The Friction Points

I don't want to make this sound like a glowing product review. There were real limitations.

Context drift. In a long conversation, Claude would sometimes forget earlier decisions — like the color variables I defined in the first message. I learned to paste in my CSS variables at the start of each new prompt so it had fresh context.

Overconfidence. A few times Claude gave me code that looked correct but had subtle issues — a missing `z-index` on a nav element, a flexbox gap that broke at a specific screen width. The outputs always needed a human eye to validate.

Design taste. AI can generate code, but it defaults to "safe" design choices. The more specific I was about aesthetics — "I want this to feel editorial, not generic startup" — the better the results. Vague prompts produce vague, forgettable designs.

What I'd Do Differently

If I were starting over, I'd build a shared style file first — colors, typography, spacing — and then build pages component by component, always referencing that file. I did this halfway through, and the second half of the project went much faster than the first.

I'd also keep a running document of my prompts. Looking back at what worked vs. what didn't would be enormously valuable the next time I use AI for a technical project.

Final Thoughts

As a Finance student, I think about AI in terms of ROI. The time I invested in learning how to prompt well — how to be specific, how to give context, how to iterate — paid dividends throughout the entire project. The code was a byproduct; the real skill was learning how to communicate precisely with a tool that can do almost anything, but only if you tell it exactly what you want.

That's not so different from managing a team, pitching to a client, or presenting to a board. Clarity of communication is the universal multiplier.
